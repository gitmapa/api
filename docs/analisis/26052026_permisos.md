## Observaciones iniciales sobre endpoints de usuarios y permisos

Se propone diferenciar los permisos de acceso a la API según perfiles de usuario:

| Perfil | Alcance general |
|---|---|
| Superadmin | Acceso total, incluyendo ABM de usuarios y administración de permisos. |
| Admin | Acceso general a datos y operaciones funcionales, excepto ABM de usuarios. |
| Analista | Acceso de lectura a endpoints definidos según necesidad de análisis. |
| Público | Acceso restringido únicamente a endpoints expresamente publicados. |

### Endpoints `users`

| Endpoint | Descripción | Perfil sugerido |
|---|---|---|
| `POST /users` | Crear un nuevo usuario | Superadmin |
| `GET /users` | Obtener todos los usuarios | Superadmin |
| `GET /users/me` | Obtener información del usuario autenticado | Todo usuario autenticado |
| `GET /users/{id}` | Obtener un usuario específico | Superadmin |
| `PUT /users/{id}` | Modificar un usuario específico | Superadmin |
| `DELETE /users/{id}` | Eliminar un usuario | Superadmin |

Se sugiere que `GET /users/me` quede disponible para cualquier usuario autenticado, ya que permite identificar la sesión activa, el rol y eventualmente los permisos disponibles para el frontend.

En caso de requerirse edición de datos propios, podría evaluarse un endpoint separado, por ejemplo `PUT /users/me`, evitando que `PUT /users/{id}` quede disponible para usuarios no administradores.
